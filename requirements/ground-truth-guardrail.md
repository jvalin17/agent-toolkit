# Requirements: Ground Truth Guardrail (G-IMPL-7)

**Status:** Draft
**Date:** 2026-05-20
**Origin:** Post-incident analysis — agent wrote test fixtures from code reading, not real data. All downstream work (implementation, demo, evaluation) built on false assumptions. Detected late.

## Problem

The agent optimizes for task completion, not correctness. When writing test fixtures, the fastest path is:
1. Read the ORM model / schema / API contract
2. Infer what the data looks like
3. Write fixtures that match the inference

This produces tests that pass, implementations that satisfy the tests, and demos that look correct — all built on invented data. The faking starts at the test fixture and propagates through every subsequent step.

**Root cause:** The agent treats its own inference as evidence. "I read the model" feels like understanding, but it's pattern completion — the agent has never seen the actual data.

## What "ground truth" means

A test fixture value has **provenance** if the agent can point to where the value came from, and that source exists outside the agent's reasoning.

### Valid provenance
| Source | Example |
|--------|---------|
| Query output | `SELECT name, email FROM users LIMIT 3` → used those 3 rows |
| API response | `curl /api/items` → used the response shape and values |
| User-provided sample | "Here's what a real record looks like: {...}" |
| Spec/contract that defines format | RFC 5322 for email format, OpenAPI schema |
| Generated by test infrastructure | Factory, Faker, builder pattern — shape from schema, values synthetic by design |
| Screenshot / UI observation | "The dashboard shows these 4 columns" |

### Invalid provenance
| Source | Why it fails |
|--------|-------------|
| "I read the model and inferred the shape" | Inference, not observation |
| "The code suggests this field exists" | Code reading, not runtime evidence |
| "This looks like what the data would be" | Pattern completion |
| "Based on the column names, I expect..." | Naming assumption |

## Requirements

### R1: Guardrail definition (G-IMPL-7)
Add to `shared/guardrails.md` under /implementation section. Hard blocker during precommit (same enforcement as G-IMPL-6).

### R2: Implementation skill — Step 0 rewrite
Current Step 0 says "show a mock of expected output." New Step 0 must require:
- If the slab touches an existing database/API/service: query it first, paste the output
- If greenfield: state explicitly that fixtures are synthetic (factory/faker) or user-provided
- The mock output must cite its source

### R3: Precommit test quality — provenance check
Add to `references/test-quality.md` audit checklist:
- Test fixtures cite their data source (query, user sample, spec, or factory)
- Flag fixtures that use specific-looking values (real dates, IDs, names) with no stated origin

### R4: Slab cycle enforcement
In the implementation slab cycle, between SETUP and TDD:
- New step: DATA — if this slab reads/writes/queries data, show evidence from the real system
- Agent must paste actual query output, API response, or user sample before writing fixtures
- "I read the schema" is explicitly called out as insufficient

## Scope boundaries

### In scope
- Guardrail text (G-IMPL-7)
- Implementation skill SKILL.md changes (step 0 + new DATA step)
- Precommit test-quality.md checklist addition
- Guardrails-quick.md one-liner

### Out of scope (parking lot)
- Structural hook that auto-detects invented fixtures at edit time (future — requires heuristics)
- CI integration that checks fixture provenance in committed tests
- Automated "run this query for me" agent behavior

## Success criteria
- Agent that follows the slab cycle is forced to show real data before writing fixtures
- Precommit catches fixtures with no provenance citation
- The guardrail text makes "I read the code" explicitly invalid as evidence
